# Задача: перевести 5 статей блога на RU и залить в Sanity с фото

**Проект:** icelab-prod (Next.js) + Sanity CMS (админка — соседний репо `../icelab-admin`).
**Цель:** взять 5 готовых UK-статей, сделать качественные RU-версии, конвертировать в Portable Text и создать документы `blogPost` в Sanity с обложками и внутренними картинками из папки фото. После — проверить, что статьи открываются на сайте на обеих локалях.

**Ничего не выдумывать: тексты уже написаны, их нужно перенести, а не переписать. RU = точный смысловой перевод UK с тем же экспертным тоном.**

---

## 1. Где что лежит

**Тексты статей (источник, UK):**
```
docs/blog/1-skilky-suhogo-lodu-potribno.md
docs/blog/2-yak-zberigaty-ta-perevozyty-suhyi-lid.md
docs/blog/3-granuly-suhogo-lodu-3-8-16-mm.md
docs/blog/4-krioblasting-ochyshchennia-suhym-lodom.md
```

Каждый файл: HTML-комментарий-шапка вверху (slug, metaTitle, metaDescription, excerpt, coverImage-плейсхолдер, author) + тело в Markdown с таблицами, ссылками и плейсхолдерами `[ФОТО: описание]`.

**Фотографии:** папка `pics-temp/` в корне проекта (~29 файлов `IMG_*.JPG`). Имена НЕинформативные — фото нужно **открыть и посмотреть глазами** (инструмент Read по картинке), затем подобрать под каждый `[ФОТО: описание]`.

**Схема Sanity (прочитать ПЕРЕД работой, не угадывать поля):**
```
../icelab-admin/schemas/documents/blogPost.ts     ← поля документа
../icelab-admin/schemas/objects/blockContent.ts   ← какие блоки/аннотации допустимы в теле
../icelab-admin/schemas/objects/tableBlock.ts      ← формат таблиц
../icelab-admin/schemas/objects/localeString.ts    ← формат {uk, ru}
../icelab-admin/schemas/objects/localeText.ts
../icelab-admin/schemas/objects/seo.ts             ← формат SEO-объекта
```
**Как рендерится тело (проверить поддержку ссылок/картинок/таблиц на фронте):**
```
src/app/components/main/Blog/PortableTextRenderer.jsx
src/sanity/queries.js   ← BODY_PROJECTION уже раскрывает internalLink (reference->slug)
```

**Пример заливки в Sanity (образец скрипта):**

Оба используют `@sanity/client` + переменные `SANITY_STUDIO_PROJECT_ID`, `SANITY_STUDIO_DATASET`, `SANITY_WRITE_TOKEN` из `../icelab-admin/.env`.

## 2. Обязательный порядок действий

### Шаг 0. Прочитать схему и рендерер
Прочитай все файлы из блока «Схема Sanity» и `PortableTextRenderer.jsx`. Зафиксируй точные названия полей и **какие типы блоков/аннотаций реально поддерживаются** (заголовки h2/h3, списки, `tableBlock`, `image`, ссылки). Дальше работай строго под этот формат.

### Шаг 1. Проверить поддержку ссылок в теле
В статьях два типа ссылок:
- **на другие статьи блога и на товары** → это `internalLink` (reference на документ Sanity). Ссылки между статьями (`/blog/<slug>`) оформляй как `internalLink` на соответствующий `blogPost`.
- **на статические страницы** (`/catalog/c/suhyi-lid`, `/catalog/c/termoboksy`, `/faq`, `/about`, `/production`, `/contacts`) и **внешние** (`https://co2lab.pro`) → это обычная ссылка с `href`.

Проверь в `blockContent.ts`, есть ли аннотация обычной ссылки (`link` с `href`). **Если её нет** — добавь её в `blockContent.ts` и обработку в `PortableTextRenderer.jsx` (внешние — `<a target="_blank" rel="noopener">`, относительные — через `next-intl` `Link`). Без этого ссылки на статические страницы и co2lab.pro работать не будут.

### Шаг 2. Конвертировать тело UK → Portable Text
Для каждой статьи преобразуй Markdown-тело в массив блоков `blockContent`:
- `##` → block style `h2`, `###` → `h3`, абзацы → `normal`.
- Маркированные списки → списки Portable Text.
- Таблицы Markdown → `tableBlock` (по формату из `tableBlock.ts`).
- Ссылки → по правилам Шага 1.
- Плейсхолдеры `[ФОТО: …]` внутри тела → блок `image` (см. Шаг 4). `[ФОТО]` из шапки — это обложка (`coverImage`).
- **Не тащи в тело служебный HTML-комментарий-шапку.**

### Шаг 3. Перевести на RU
Сделай `bodyRu` — точный смысловой перевод `bodyUk`. Требования:
- Сохрани экспертный тон, все цифры, таблицы, ссылки и структуру 1:1.
- Соблюдай анти-AI-slop (активный залог, конкретика, без штампов и без декоративных тире). При наличии — прогони через скилл `stop-slop`.
- Термины единообразно: «сухий лід» → «сухой лёд», «сублімація» → «сублимация», «гранула» → «гранула», «термобокс» → «термобокс», «рекуперація» → «рекуперация».
- Переведи и `title`, `excerpt`, `metaTitle`, `metaDescription`, и alt-тексты картинок.

### Шаг 4. Подобрать и загрузить фото
1. Посмотри изображения в `pics-temp/` (Read по каждому файлу).
2. Под каждый `[ФОТО: описание]` подбери максимально подходящий кадр. Логика по статьям:
   - №1 (расчёт) — весы, зважування, мерный совок с гранулами.
   - №2 (хранение/перевозка) — термобокс, гранулы в боксе, погрузка.
   - №3 (гранулы) — крупные планы гранул разного размера.
   - №4 (криобластинг) — процесс очистки/аппарат/деталь (если нет — общий план льда/оборудования).
   - №5 (производство) — цех, линия, выход гранул, склад.
   Один кадр не использовать в двух статьях, если есть выбор.
3. Загрузи выбранные файлы как ассеты: `client.assets.upload('image', fs.createReadStream(path))`.
4. `coverImage` = `{_type:'image', asset:{_type:'reference', _ref: <assetId>}, alt: {uk, ru}}`, где alt = текст из плейсхолдера (переведи на обе локали).
5. Внутренние `[ФОТО]` → блоки `image` в теле с тем же принципом alt.
> Файлы крупные (2–6 МБ). Заливать оригиналы можно — Sanity CDN + `next/image` отдадут оптимизированные версии. EXIF-ориентацию проверь, чтобы кадры не были повёрнуты.

### Шаг 5. Создать документы в Sanity
Напиши Node-скрипт (по образцу `create-food-ice-category.mjs`) в `../icelab-admin/`, запускай оттуда (там лежит `.env` с токеном):
- `_id` — детерминированный, напр. `blogPost-<slug>`, и используй `createOrReplace` (идемпотентно, повторный запуск не плодит дубли).
- Поля: `title{uk,ru}`, `excerpt{uk,ru}`, `slug.current`, `coverImage`, `publishedAt`, `bodyUk`, `bodyRu`, `seo` (в `seo.title{uk,ru}` — metaTitle, `seo.description{uk,ru}` — metaDescription; остальные поля seo — по схеме `seo.ts`).
- `publishedAt` — разнеси по датам (например, 5 разных дней за последние 2–3 недели), чтобы список выглядел естественно, а не одной пачкой.
- В `blogPost` **нет поля категории** — не добавляй.

### Шаг 6. ⚠️ Статья №2 заменяет существующую
В датасете уже есть пост со slug `yak-zberigaty-sukhyi-lid` (черновик). Новая статья №2 (`yak-zberigaty-ta-perevozyty-suhyi-lid`) его **замещает по смыслу**. Чтобы не было двух страниц под один запрос:
- Найди старый документ (`*[_type=="blogPost" && slug.current=="yak-zberigaty-sukhyi-lid"]`).
- **Удали его** (или переопredели его slug на новый и перезапиши контентом новой статьи). В датасете должен остаться **один** пост про хранение.

- [ ] В датасете 5 постов (не 6): старый `yak-zberigaty-sukhyi-lid` удалён/переопределён.
- [ ] Запусти сборку `icelab-prod` (`npm run build`) — зелёная.
- [ ] Открой `/blog` и каждую `/blog/<slug>` в **uk и ru**: тело рендерится, таблицы на месте, картинки грузятся, ссылки (внутренние, статические, co2lab.pro) кликабельны.
- [ ] Нигде не осталось `[ФОТО:` и слова `TODO`.
- [ ] На странице статьи есть `BlogPosting` JSON-LD, у обложки заполнен alt.

5 статей опубликованы в Sanity (uk+ru), с обложками и внутренними фото из `pics-temp/`, корректными таблицами и ссылками; старый дубль про хранение удалён; сайт собирается; все страницы блога открываются на обеих локалях без плейсхолдеров. В финале — короткий отчёт: какие фото под какую статью пошли, какие даты публикации проставлены, что сделано со старым постом.
